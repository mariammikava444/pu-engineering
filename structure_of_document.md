# მოთხოვნების დოკუმენტის სტრუქტურა
___
**მოთხოვნების დასაზუსტებლად დღეს ყველაზე ხშირად ბუნებრივი ენები გამოიყენება.**
თუმცა SRS-ის რომელიმე ფრაგმენტი შეიძლება იყოს წარმოდგენილი სპეციფიკური ენით. როდესაც ფორმალური ენები გამოიყენება, ისინი ხშირად გამოიყენება კონკრეტული თვისებების ან სისტემის კონკრეტული ნაწილების დასაზუსტებლად, როგორც მთლიანი SRS-ის ნაწილი.
IEEE-ს სტანდარტის მიხედვით, სხვადასხვა პროექტმა შეიძლება მოითხოვოს მათი მოთხოვნების განსხვავებულად ორგანიზება, ანუ არ არსებობს ერთი მეთოდი, რომელიც შესაფერისია ყველა პროექტისა და მოერგება ყველანაირ პროექტს. **IEEE გვთავაზობს SRS-ის სტრუქტურირების სხვადასხვა გზებს.**
# General Structure of an SRS
___
1. Introduction
	1. Purpose
	2. Scope
	3. Definitions, Acronyms, and Abbreviations
	4. References
	5. Overview
2. Overall Description
	1. Product Perspective
	2. Product Functions
	3. User Characteristics
	4. General Constraints
	5. Assumptions and Dependencies
3. Specific/Functional/Detailed Requirements
## Introduction
შეიცავს მოთოხვნების დოკმენტის მიზანს, სფეროს, მიმოხილვას და ა.შ.
მთავარი ასპექტი არის მოტივაციისა და ბიზნეს მიზნების გარკვევა, რომლებიც განაპირობებს ამ პროექტს და პროექტის სფეროს.
### Overall Description
მოცემულია სისტემის საერთო პოზიციონირება (perspective) - როგორ ჯდება ის უფრო დიდ სისტემაში და ამ სისტემის ყველა მოთხოვნის მიმოხილვა.
**ამ დროს დეტალური მოთხოვნები არ არის ნახსენები!**
პროდუქტის პოზიციონირება (product perspective) არსებითად არის პროდუქტის ურთიერთობა სხვა პროდუქტებთან; განსაზღვრავს, არის თუ არა პროდუქტი დამოუკიდებელი ან არის თუ არა იგი უფრო დიდი პროდუქტის ნაწილი და რა არის პროდუქტის ძირითადი ინტერფეისები.
### Specific/Detailed Requirements
დეტალური მოთხოვნების სექცია აღწერს იმ მოთხოვნების დეტალებს, რომლებიც დეველოპერმა უნდა იცოდეს სისტემის დიზაინისა და განვითარებისთვის.
მოთხოვნები შეიძლება ორგანიზებული იყოს მუშაობის რეჟიმის (modes of operation), მომხმარებლის კლასის (user class), ობიექტის (object), მახასიათებლის (feature), სტიმულის (stimulus) ან ფუნქციური იერარქიის (functional hierarchy) მიხედვით.
კონკრეტული მოთხოვნების ორგანიზების ერთ-ერთი მეთოდია ჯერ გარე ინტერფეისების დაზუსტება, რასაც მოჰყვება ფუნქციური მოთხოვნები, წარმადობის (performance) მოთხოვნები, დიზაინის შეზღუდვები და სისტემის ატრიბუტები…
### Functional Requirements
**ამ სექციაში მოცემულია პროგრამული უზრუნველყოფის მთლიანი სისტემის ყველა ფუნქციონალური შესაძლებლობა.** 
თითოეული ფუნქციური მოთხოვნისთვის უნდა იყოს მითითებული საჭირო შემავალი და გამომავალი მონაცემები და შესასრულებელი ოპერაციები.
შემავალი მონაცემისთვის (input-ისთვის) უნდა იყოს მითითებული წყარო, საზომი ერთეულები, მოქმედი დიაპაზონები, სიზუსტეები და ა.შ.
### Requirements about Performance
**წარმადობის (performance) განყოფილება** უნდა იყოს მითითებული როგორც სტატიკური, ასევე დინამიური წარმადობის მოთხოვნები.
ყველა ფაქტორი, რომელიც ზღუდავს სისტემის დიზაინს, აღწერილია წარმადობის შეზღუდვების განყოფილებაში.
# Use Cases
___
**Use case**-ები აღწერს სისტემის ქცევებს, როგორც მომხმარებლის ურთიერთქმედებებს სისტემასთან.
Use case-ები შეიძლება გამოვიყენოთ დიდი ბიზნესის ან ორგანიზაციის ბიზნეს პროცესების აღსაწერად, რომელიც იყენებს პროგრამულ უზრუნველყოფას, ან უბრალოდ პროგრამული სისტემის ქცევის აღსაწერად.

**Use Case Terms**:
- **Actor** - A person or a system which uses the system being built for achieving some goal.
- **Primary actor** - The main actor for whom a use case is initiated and whose goal satisfaction is the main objective of the use case.
- **Scenario** - A set of actions that are performed to achieve a goal under some specified conditions.
- **Main success scenario** - Describes the interaction if nothing fails and all steps in the scenario succeed.
- **Exception scenario** - Describes the system behavior if some of the steps in the main scenario do not complete successfully.
### Actor
Use case ტერმინოლოგიაში, მოქმედი პირი (იგივე აქტორი) არის ადამიანი ან სისტემა, რომელიც იყენებს სისტემას გარკვეული მიზნის მისაღწევად.
### Primary Actor
მთავარი მოქმედი პირი (მთავარი აქტორი) არის მთავარი პირი, რომელიც აღძრავს use case-ს მიზნის მისაღწევად და რომლის მიზნის დაკმაყოფილება არის use case-ის მთავარი მიზანი.
### Scenario
ინტერაქციის აღწერისთვის use case-ები იყენებს **სცენარებს**.
სცენარი აღწერს მოქმედებების ერთობლიობას, რომლებიც უნდა შესრულდეს კონკრეტული მიზნის მისაღწევად, გარკვეულ პირობებში.
სცენარის თითოეული ნაბიჯი (step) არის ერთი სრულყოფილი ქმედება, რომელსაც ასრულებს ან აქტორი ან სისტემა. ზოგადად, ნაბიჯი არის აქტორის გარკვეული ქმედება (მაგ. ინფორმაციის შეტანა), ზოგიერთი ლოგიკური ნაბიჯი (მაგ. ინფორმაციის გადამოწმება, ინფორმაციის მიწოდება) ან სისტემის შიდა მდგომარეობის ცვლილება, რომელსაც სისტემა ასრულებს თავისი მიზნების მისაღწევად (მაგ., ტრანზაქციის რეგისტრაცია, ჩანაწერის განახლება).
### Main Success Scenario
Use case-ს ყოველთვის აქვს მთავარი წარმატებული სცენარი, რომელიც აღწერს ინტერაქციას, თუ არაფერი "ჩავარდა" და სცენარის ყველა ნაბიჯი წარმატებულია.
### Exception Scenario
ამ სიტუაციებისთვის, use case-ს აქვს გამონაკლისი სცენარები, რომლებიც აღწერს სისტემის ქცევას იმ შემთხვევაში, როდესაც ძირითადი სცენარის ზოგიერთი ნაბიჯი წარმატებით არ დასრულდება.
___
**Use case-ები იძლევა იერარქიულ ორგანიზებას.**
მაგალითად, უნივერსიტეტის აუქციონის სისტემისათვის გვექნება შემდეგი use case-ები:
```
UC1 : "Put an item for auction",
UC2 : "Make a bid",
UC3 : "Complete and auction of an item".
```
### UC1: Put an item for auction
___
**Primary Actor**: Seller

**Precondition**: Seller has logged in

**Main Success Scenario**:
1. Seller posts an item (its category, description, picture, etc.) for auction
2. System shows past prices of similar items to seller
3. Seller specifies the starting bid price and a date when auction will close
4. System accepts the item and posts it

**Exception Scenarios**:
1. There are no past items of this category
2. System tells the seller this situation

# ER (Entity Relationship Diagrams)
___
**ER დიაგრამა გამოიყენება სისტემის მონაცემების მოდელირებისთვის და იმის საჩვენებლად თუ როგორ კავშირში არიან ერთმანეთთან ისინი**, მაგრამ იგი არ განიხილავს იმ საკითხს თუ თავად ეს მონაცემები როგორ მუშავდება. ხშირად იგი გამოიყენება მონაცემთა ბაზებში ცხრილებისა და მათი ერთმანეთისადმი დამოკიდებულების (რელაციის) აღსაწერად.

